# P6 ŌĆö Swarm + DPad Menu

One player drone + N physics-based followers (default 5, `@export`-variable),
formation flight via a dedicated flight mode, a dpad menu UI, then ŌĆö once the
menu exists ŌĆö formation cycling, auto-land, and crosshair dispatch.

Pushed to P7 (too much for one phase, thematically distinct): payload physics +
DELIVER targets, visible interceptor, quest upgrades, adversary drones. Marked
as P7 candidates in `backlog/`.

## Hard constraints

- Followers are full `DroneController` physics drones ŌĆö rotor-only forces, no
  magic ŌĆö so future flight-controller work (battery etc.) applies swarm-wide.
- Jamming must disrupt followers analogously to the player: degraded "signal
  receive" of the leader's position, plus the controller's own per-drone
  `lose_signal()` (already generic).
- Tim flight-tests in-editor at each milestone before the next begins.

## Established facts

- `_read_inputs()` (`drone_controller.gd:362`) is the only stick-input read
  point; everything downstream (mode ŌåÆ mixer ŌåÆ rotor forces) is input-agnostic.
- WindField is already per-drone (each controller samples wind at its own
  position, `drone_controller.gd:503`). SignalField / MissionTarget /
  MissionTracker are group-based and swarm-ready.
- Single-drone `drone_path` assumptions to fix: `chase_camera.gd`,
  `debug_hud.gd`, `flight_recorder.gd`, `airspace_control.gd`,
  `crash_effects.gd`.
- Dpad + all four face buttons are unbound. Godot `JoyButton` is SDL-based and
  controller-agnostic: `JOY_BUTTON_DPAD_*` (11ŌĆō14), face buttons `A/B/X/Y` =
  Cross/Circle/Square/Triangle. No hat-axis trap (unlike the L2/R2 one).

## Workstreams

### 1. Multi-drone groundwork

- `@export var is_player := true` on DroneController; joins group
  `"player_drone"` when set (all drones stay in group `"drone"`).
- Convert the 5 `drone_path` consumers to lazy `"player_drone"` group lookup
  (WindField pattern), keeping `drone_path` as explicit override.
  `AirspaceControl` + `CrashEffects` watch **all** of group `"drone"`;
  camera / HUD / recorder stay player-only.
- `MissionTarget` observe/crash: iterate group `"drone"` so followers can
  clear targets.

### 2. Formation flight mode + follower pilots

- `scripts/drone/flight_mode_formation.gd` ŌĆö a `FlightModeBase` mode computing
  `FlightControl` directly from a target position + heading (position error ŌåÆ
  tilt, altitude error ŌåÆ collective, heading error ŌåÆ yaw torque), ignoring the
  stick args. Rationale: feeding stabilized mode breaks down when the leader
  flies acro; computing from target position is stable whenever the target
  path is. Gains as `@export` tuning knobs.
- `scripts/swarm/follower_pilot.gd` ŌĆö one pilot per follower. Behavior states
  (`FORMATION` / `HOLD` / `DISPATCHED` / `LANDING` / `DOWN`), updates the
  formation mode's target each tick, behavior switched via signals.
  **Signal receive:** pilot samples `SignalField.get_quality()` at the
  follower's position; packet loss freezes the *target data* (stale leader
  position), mirroring how player packet loss freezes stick inputs.
- `scripts/swarm/swarm_manager.gd` ŌĆö thin: group `"swarm_manager"` (absent =
  no swarm), `@export var follower_count := 5`, spawns `drone.tscn` + pilot
  pairs, owns the formation slot table.
- Spawn: airborne, already in slots above the pad. Grounded takeoff
  sequencing is a follow-up once the formation mode is trusted.
- Formations: slot tables relative to leader pose: `LINE`, `V`, `RING`; then
  `BOHR` (orbiting electron-cloud ŌĆö slots rotate over time, same controller).

### 3. DPad menu + FPV cam pitch

- InputMap (controller-agnostic JoyButton + keyboard fallbacks): `menu_open`
  (dpad left/right), `menu_up` / `menu_down` (navigate), `menu_confirm`
  (Cross), `menu_back` (Circle).
- Outside menu mode: dpad up/down adjusts FPV camera pitch (export on
  `chase_camera.gd`). Menu mode captures the dpad until closed.
- `scripts/ui/pad_menu.gd` ŌĆö Forza-style strip, hand-built Controls in
  debug_hud style. Entries as a data array `{label, get, adjust/confirm}` so
  later entries (targeting mode, payload in P7) plug in without rework.
- P6 entries: Formation (LINE/V/RING/BOHR), Auto-land, Recall/Hold.
- While CRASHED the menu is dead like all other input.

### 4. Formation cycling, auto-land, dispatch (closes P6)

- Formation cycling + auto-land as menu entries. Auto-land: pilots ramp toward
  `terrain.get_height()` + settle to idle ŌĆö a throttle ramp, not a teleport.
- Crosshair dispatch: reticle drawn at the **ground-projected raycast hit**
  from screen center (VTOL VR-style ŌĆö a fixed center reticle makes aiming
  under perspective miserable); highlight when a `MissionTarget` is within
  pick radius; **Square** (`dispatch_follower`, its own button ŌĆö no menu
  overloading) dispatches the nearest FORMATION follower. It flies to the
  target, clears it per type (hover OBSERVE, impact CRASH), returns to
  formation or stays DOWN. Jamming en route degrades it via signal-receive +
  the controller's own signal loss.
- Later phase: movable-crosshair "targeting mode" as a menu entry ŌĆö the
  data-driven menu + raycast reticle keep that door open.
