# P6.6 — Polishing Pass Implementation Plan

## Context

P6.5 shipped the swarm QoL batch. Before the P7 gameplay/missions push, [P6.6-shortlist.md](plans/P6.6-shortlist.md) collects tweaks to existing systems: the long-awaited stabilized-mode PID fix (blend the two hard-switched control laws), a smarter kamikaze run-in, prop-obstruction physics, HUD theme extraction, and an FPV attitude indicator. Suggested order (each item = one conventional commit):
1. PID blend → 2. kamikaze approach → 3. prop obstruction → 4. HUD theme extraction → 5. attitude indicator.

**Workflow gate:** after implementing each item, **stop and hand over to the user for in-editor validation** before committing or starting the next item. No commit happens until the user has flown/checked the change themselves.

## 1. PID fine-tuning — blend the stabilized control laws

**File:** `scripts/drone/flight_mode_stabilized.gd`

**Diagnosis (per AGENTS.md Known Issues + code):** two separate defects.

- **"Jump" on stick release:** rate mode (`rate_p_gain = 4.0`) and auto-level (`stabilize_p_gain = 15.0`, ~4× stronger) are hard-switched at `input_deadzone = 0.05` (line 71–75). Releasing the stick while tilted snaps into the much stiffer angle law.
- **"Sticky" near level under stick input:** rate mode's feedback reads `_filtered_ang_vel` (one-pole IIR, `gyro_filter_alpha = 0.35`, line 65/68), adding lag that is proportionally worst for small commanded rates.

**Fix (compute-both-and-blend):**

- Compute **both** torque outputs every frame: `rate_torque` (rate PD) and `level_torque` (angle P + filtered-gyro D).
- Blend weight from stick deflection: `s = max(|pitch|, |roll|, |yaw|)`, `w = smoothstep(input_deadzone, input_deadzone + blend_band, s)` with new `@export var blend_band: float = 0.2`. Output `torque = level_torque.lerp(rate_torque, w)`; `yaw_torque = w * yaw * 1.5` (auto-level contributes 0 yaw, as today).
- **Separate gyro signals:** rate mode's `rate_error` uses the **raw** `angular_velocity` (body-frame); the low-pass `_filtered_ang_vel` stays exclusively on the auto-level D term. No new filter knob — raw feedback is the standard rate-loop input; `gyro_filter_alpha` remains the tuning knob for the D path.
- Existing `max_offset` clamp and collective handling unchanged. Update the class doc comment + the two Known Issues entries in AGENTS.md / PROJECT_SUMMARY.md (both issues are addressed by this change).

**Tests:** all 15 existing tests in `scripts/test/flight_mode_test.gd` must pass unchanged (they assert self-leveling, input response, bounded coupling). Add one pure test calling `compute()` directly with stick at `input_deadzone ± ε` and asserting the output `pitch_diff` is continuous (delta below a small epsilon) — that's the regression test for the hard switch. Feel-tuning of `blend_band` is manual (user flies it).

## 2. Better kamikaze approach — glideslope run-in

**Files:** `scripts/swarm/follower_pilot.gd` (main), `scripts/drone/flight_mode_formation.gd` (unchanged or minimal).

**Current behavior:** `_dispatch_aim()` (follower_pilot.gd:128–142) cruises at the captured `_cruise_agl` until within `dive_radius = 12 m` horizontally, then `_fly_dispatch()` latches `_plunging` and `_strike()` points full collective at the target — so the drone must null its horizontal speed overhead before the dive verticalizes.

**Fix (aim-point shaping, no new control law):**

- In the kamikaze branch of `_dispatch_aim()`, replace the flat cruise altitude with a **glideslope**: `aim_alt = min(cruise_alt, target.y + horiz_dist * tan(dive_angle))`, with `@export var dive_angle_deg: float = 35.0` on FollowerPilot. Beyond the slope-intercept distance this is identical to today's cruise; inside it the follower descends toward the target while still moving horizontally.
- Feed the slope into the existing velocity feed-forward: set `_mode.target_velocity` along the descending approach direction (unit vector toward the glideslope aim, scaled by current approach speed) so the position cascade doesn't brake at the aim point. The formation mode's cascade (`ff_h` + altitude D feed-forward, lines 113–149) already consumes `target_velocity` — no mode changes needed for this part.
- **Earlier, alignment-based strike commit:** in `_fly_dispatch()` (lines 300–309), commit when the 3D distance to the target is inside `dive_radius` **or** the drone's velocity points within ~25° of the target line while descending and inside ~2× `dive_radius`. The strike phase itself stays as-is — unconstrained attitude, upside-down hits remain feasible; the win here is purely approach geometry (no hover-and-null-speed overhead before committing). `STRIKE_TIMEOUT` safety net (3 s → `lose_signal()`) stays untouched.

**Tests:** add to `scripts/test/swarm_test.gd` (pattern: build `SwarmManager` in code, double-await spawn, run physics ticks): dispatch a follower at a `MissionTarget.Type.CRASH` placed ~60 m away; assert (a) it crashes/clears the target within a time budget, and (b) the flight-path angle is meaningfully steep before commit — e.g. descent rate ≥ tan(20°) × horizontal speed sampled just outside `dive_radius`. (A bare "vertical velocity is negative" check could plausibly already pass on the current code from terrain-following wobble — assert the slope, not just the sign. Run the new test against the unmodified code first to confirm it fails.)

## 3. Prop obstruction — clipped props lose thrust

**Files:** `scripts/drone/drone_controller.gd` (+ small doc updates).

**Design:** per-rotor shape query each physics tick; an obstructed rotor contributes zero thrust, so the remaining rotors' asymmetric forces tumble the airframe naturally — rotor-only, no magic forces. Strong impacts on a prop disable that rotor until reset.

- Add state: **one array**, `var _prop_state: Array[int] = [0, 0, 0, 0]` — `0` = free, `1` = obstructed (transient, recomputed every tick from the query), `2` = broken (latched until `reset()`). Plus one shared `SphereShape3D` query (prop disc radius ≈ 0.12 m — verify against the actual GLB prop mesh extent) via `PhysicsShapeQueryParameters3D`, excluding the drone's own RID.
- In `_physics_process` (or at the top of `_compute_and_apply_forces`), run `direct_space_state.intersect_shape()` at each of the 4 world-space rotor positions (`global_transform.basis * _rotor_positions[i] + global_position`). For each rotor with `_prop_state[i] != 2`: set to `1` on hit, `0` otherwise.
- In the force loop (drone_controller.gd:429–438): `if _prop_state[i] != 0: continue` (skip `apply_force` for that rotor; zero that entry in `last_mix` so telemetry shows reality). Yaw torque: scale by the fraction of live rotors so a dead prop also degrades yaw authority.
- **Permanent break (state 2):** when a rotor transitions 0→1 while its commanded throttle > threshold (e.g. 0.5) **and** relative impact speed `_prev_velocity.length()` > a `prop_break_speed` export (e.g. 3 m/s), latch `_prop_state[i] = 2`; `reset()` clears the whole array to 0. Visual: swap that rotor's blur disc back to the idle mesh via the existing `_rotor_nodes` / `_rotor_spin_materials` arrays (same swap as `_set_armed()`, lines 519–529). Log one HUD event line ("PROP n OUT") via the existing event-log path if trivially reachable, otherwise skip.
- Doc: add a Known-Issues-style note that obstruction is a hub-sphere approximation, not per-blade.

**Tests:** in `flight_mode_test.gd` (or a new small suite reusing its scene): place a `StaticBody3D` box overlapping one rotor position, apply collective, run ticks, assert the drone develops angular velocity tipping toward the obstructed corner while a clean drone at the same throttle stays level. Plus a pure check that a state-2 (broken) prop survives until `reset()` clears it.

## 5. UI attitude indicator (instrument-style, FPV-only)

**Files:** `scripts/ui/debug_hud.gd`, `scripts/ui/pad_menu.gd`.

Reuses the overlay-canvas drawing mechanics (`draw` signal + `_on_*_draw()`, like the dispatch reticle at debug_hud.gd:274–289) but **not** the panel styling: **no background ColorRect, no green-on-black** — bare white line work drawn directly at screen center, like real FPV goggle OSDs.

- `_attitude_canvas` (Control, `PRESET_FULL_RECT`, `MOUSE_FILTER_IGNORE`), `draw` → `_on_attitude_draw()`, `queue_redraw()` in the existing per-frame block (~line 211).
- Draw (white `Color(1,1,1,α)`, thin lines — classic instrument):
  - Fixed center reference: small winged-W / dash-dot-dash at screen center.
  - Horizon line rotated by −roll around center, offset vertically by pitch × px-per-degree (~4 px/°), drawn with the wind-arrow-style dark outline for readability against sky.
  - Pitch ladder: ±10/20/30° rungs with degree labels, rotated with the horizon, clipped to a max radius so it stays a compact instrument rather than full-screen clutter.
- Pitch/roll from `_drone.global_transform.basis.get_euler()` (`euler.x` = pitch, `euler.z` = roll) — same source the telemetry label uses (~line 401). Camera tilt deliberately ignored (instrument-style).
- Gate on `_drone.is_fpv_enabled()` at the top of the draw callback (same as the dispatch reticle) and on a new `show_attitude: bool = true` property with the visibility-setter pattern of `show_log`/`show_gizmo` (debug_hud.gd:92–117).
- Add an `ATTITUDE` entry to `_build_hud_entries()` in pad_menu.gd (lines 125–181) — copy the GIZMO toggle's getter/setter shape.

**Verification:** manual — user flies FPV with cam tilt and checks readability (per established workflow, in-editor feel testing is theirs). Headless suites just need to keep passing.

## 6. Split PROJECT_SUMMARY.md into semantic docs

**Problem:** `PROJECT_SUMMARY.md` is ~860 lines and growing — the per-script deep-dives and full tuning table mean you re-read the whole file to find one system. Split so each concern lives in its own file and the summary stays a scannable index.

**Target layout** (reuse the existing `docs/` folder — `docs/publishing.md` already lives there):

- **`PROJECT_SUMMARY.md` keeps** (high-level only): intro/current-phase, Scene Tree, the Three-Layer Flight Pipeline overview, a one-line-per-script architecture map, Coordinate System, Flight Model overview, and a compact **exposed tuning knobs** summary (the live-tunable `@export` groups — Stabilized Gains, Formation Gains — with default + range, not the exhaustive internal table). Every section that moves out gets a one-line pointer + link to its new file.
- **New `docs/systems/` files** carry the detail (per-script `### scripts/...` deep-dives + their internal params), split by the summary's existing top-level sections so the boundaries are already drawn:
  - `docs/systems/flight.md` — drone_controller core, flight_mode_base, acro, stabilized, altitude_hold, brake, the flight pipeline detail, wind drag.
  - `docs/systems/environment.md` — wind_field, wind_particles, crash_effects, camera, debug_hud (telemetry/wind arrow).
  - `docs/systems/terrain.md` — Real-World Terrain (P4): bake tool + OsmTerrain runtime.
  - `docs/systems/gamification.md` — Gamification (P5): signal field, post shader, radar, compass, mission targets/tracker, jamming.
  - `docs/systems/swarm.md` — Swarm (P6): swarm_manager, flight_mode_formation, follower_pilot, pad_menu, dispatch reticle.
  - `docs/systems/tuning.md` — the full Tuning Parameters table (all internal params), plus the per-test-harness notes if they don't fit elsewhere.

**Approach:** pure content move — cut each detailed section verbatim into its new file (keep headings), leave a linked stub in the summary. No prose rewrites beyond the stub pointers. Update the cross-references: AGENTS.md and CLAUDE.md both say "PROJECT_SUMMARY.md is the deep-dive reference" — point those at the new `docs/systems/` set. Grep for `PROJECT_SUMMARY.md →` mentions across the repo (AGENTS.md uses that phrasing a lot) and repoint each to the file that now owns that section.

**Verification:** `grep -rn "PROJECT_SUMMARY.md" .` finds no dangling section references; every `docs/systems/*.md` link resolves; the summary reads top-to-bottom as an index. No code touched, so tests are unaffected — a `./run_tests.sh` sanity pass is optional. This is the largest single doc edit; consider its own commit (`docs: split PROJECT_SUMMARY into per-system reference files`).
